## General Engagement Scope

Microsoft will support our data protection commitments in the following areas:

*Use the following guidance when documenting scope from this general engagement scope table:*
Be sure to refer to PPBS Mapping Worksheet for finding applicability of the below listed activities in the table.
*TAILOR OUT THE ACTIVITIES WHICH ARE NOT APPLICABLE FOR THE ENGAGEMENT.*
Refer to the [Primary Product/Business Scenario Mapping Worksheet](https://aka.ms/securityestimator) for applicability.


| Area | Description | Assumptions |
| :-- | :-- | :-- |
| Data Protection Questionnaire (DPQ) | - Conduct discovery of <customer name>'s data protection environment for engagement delivery. | - Customer provides baseline understanding of their relevant data protection requirements.<br> - Customer provides inputs for data classification. <br>- Complete a Microsoft-internal DPQ covering discovery, classification, and control applicability. |
| Solution Security Design Review | - Review/assess solution architecture for design-related security issues.<br>- Recommend mitigations for identified issues. | - Customer will provide relevant security policy documentation.<br>- Customer SME participates in the review. |

## Out of Scope

Review and update the optional "out of scope" items if Security Service Line services are included in the engagement.


| Area | Description |
| :-- | :-- |
| Information security policies | Creation of information/application development security policies is out of scope. |
| Application security code review | No security review will be conducted for applications outside the scope of this engagement. |
| Comprehensive security and compliance assessment | Microsoft review is limited to features within scope and isn’t a full marketplace/environment review. |
| Regulatory Standards | Attestation/certification of the application to any regulatory standards is out of scope. |

## Approach

**Important:**
The following table section includes sample activities that are common to most components. Revise as necessary/applicable.

The delivery compliance, privacy and security strategy will be pursued by implementing the following process steps in the engagement, which describe objectives and outcomes of each step.

Below is a diagram showing the high-level approach timeline and how activities are performed:

### Flowchart: Engagement Approach

- Data Privacy Review
- Engagement Initiation
- Product Baseline Planning
- Delivery Sprints
- Data Protection in Sprints
- Engagement Start
- Engagement End

***

### Delivery Sprints

In each delivery sprint, the following activities are performed:


| Category | Description |
| :-- | :-- |
| **Microsoft:** | - Conduct discovery of baseline requirements via the DPQ.<br>- Identify issues/compliance requirements based on customer info.<br>- Review solution design, document/report findings. |
| **Customer:** | - Provide Microsoft needed security documentation.<br>- Provide architecture/integration documents<br>- Attend classification/design review sessions<br>- Review product for conformance. |

#### Outcomes

| Name | Description | Acceptance required? | Responsibility |
| :-- | :-- | :-- | :-- |
| Security Findings | Security findings may be shared with customer and entered into project planning/risk log/task backlog as applicable. | No | Microsoft |


***

### Engagement End

At the end of the engagement, these activities are performed:


| Category | Description |
| :-- | :-- |
| **Microsoft:** | - Data protection clean-up, remove customer data.<br>- Hand off all credentials/secrets relevant to scope. |
| **Customer:** | - Remove Microsoft team access to all systems.<br>- Rotate/change credentials/secrets.<br>- Review roles, permissions, configurations. |
